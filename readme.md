## 使用建议

1. 大多数软件开发任务，先看 [软件开发主流程](./docs/software-development.md)，再按需要跳到 [语言与框架生态](./docs/languages-and-frameworks.md)。
2. 不要先按来源仓库找 skill，先按你当前遇到的问题类型找目录。
3. 如果你在处理项目文档体系、AI 首屏入口、canonical source，或者总览页该怎么组织，先看 [软件开发主流程](./docs/software-development.md) 里的文档结构、治理与上下文治理。
4. 如果你主要在改措辞、压句子、统一术语，或者把草稿重写成更清楚的说明，先看 [元技能与辅助](./docs/meta-skills.md) 里的文档措辞、重写与风格。
5. 如果一个 skill 同时看起来像“流程工具”和“生态工具”，优先以它最常出现的使用场景为准。

## 说明

这份指南按“查阅频率 + 内容性质”组织：

- 高强度开发时会反复查的主流程，集中放到一个文件里。
- 体量大的语言 / 框架生态，单独拆成一个文件。
- 和开发主流程差异明显的思维模型，单独拆成一个文件。
- Three.js / Web3 / Slidev 这类专项领域，单独拆成一个文件。
- 文档结构与治理放在主流程里，因为它通常和需求、方案、协作、交付一起出现。
- 文档措辞与重写放在元技能里，因为它更像跨场景复用的辅助能力。

这样可以同时满足两件事：

- 开发时不用在太多文件之间来回跳。
- 内容继续增长时，不会反复打乱主目录结构。

## 格式规范

### 分组

- 同一个包 / 仓库下有多个 skill：使用 `### 📦 [包名](链接)` 分组
- 只有单个 skill：直接写条目，不单独起 `📦`
- 没有稳定上游来源时：标注 `（本地技能）`
- 一旦能确认上游 GitHub：优先写 GitHub，不再只写“本地技能”
- 对体量较大、成体系、值得单独维护的外部技能集：单独建 `pack-*.md` 文件，不直接混入主正文结构

### 条目格式

```md
- **`skill-name`** ([source](https://github.com/...))
    - **介绍**：它是什么，核心解决什么问题。
    - **场景**：什么情况下应该想到它。
    - **组合**：通常和哪些 skill 连用，或它和相近 skill 的分工。
    - **补充**：这个 skill 特有的注意点、行为方式或限制。
```

### 字段规则

- `介绍`：必填
- `场景`：推荐填写；对含义非常明确的基础库型条目可以省略
- `组合`：按需填写；只有存在稳定搭配、前后关系或边界区别时才写
- `补充`：按需写
- 字段顺序固定为：`介绍` → `场景` → `组合` → `补充`
- 同一条里只写真正有信息量的字段，不为了凑格式硬补空话
- 如果 skill 有上游来源，来源信息要直接写在 skill 行里，不能只依赖分组标题
- 如果一个 skill 需要表达特殊信息，优先放进 `补充`，不要随意新增同级字段
- 只有当一类特殊信息在多条 skill 中反复出现时，才考虑新增正式字段

### 字段含义

- `介绍`：定义 skill 的本质，不重复 skill 名，不写“这是一个很强大的工具”这类空话
- `场景`：回答“什么时候该想到它”，优先写触发条件、问题特征或项目状态；对像 `pinia`、`vueuse-functions` 这类含义极明确的基础库型条目，可以更短，甚至省略
- `组合`：说明它和其他 skill 的前后关系、分工边界、替代关系或典型搭配；没有稳定组合时可以省略
- `补充`：承载不适合放进前三项的信息，比如特有工作方式、限制、注意点、默认倾向，或需要特别提醒的上下文

### 额外要求

- 条目要能独立成立，不依赖分组标题才能看懂来源
- `介绍` 是底线；`场景`、`组合`、`补充` 都不为了凑格式而写
- 相似 skill 尽量写出边界，不要只写“都很好用”
- 中文名只在确实有帮助时加，不强制每条都起
- 单条描述尽量短，但要让读者看完后能直接判断“要不要用”
